## Audit Scope

The repository is large enough to require focused passes rather than a claim of line-by-line review: roughly 40k TypeScript lines across `packages` and `scripts`, excluding generated `dist` and dependencies. Review source in the current working tree, use prior audit notes only to avoid re-adding completed items, and keep this file limited to actionable current debt.

The timeout-related work must start with the shared policy and diagnostics contract before editing individual loops. Implement the central timeout policy first, then wire native frame deadlines, Marionette framing and pending cleanup, browser wait cancellation, process-stop escalation metadata, and eval/wait execution semantics to that policy. This avoids adding new one-off constants while fixing the current one-off constants.

## High Workstreams

- [x] Consolidate the remaining CLI argv parsing sources into typed route contracts. `packages/cli/src/route-registry.ts` centralizes route ids, help, parser flags, formatter choice, and builder binding, but request builders in `packages/cli/src/commands/*.ts` still reparse argv through independent helpers in `packages/cli/src/parse.ts`, and `parse.ts` keeps separate global option classifications in `isBooleanPositionalOption` and `isValuePositionalOption`. Replace the route parser specs, positional helper option lists, per-command parser loops, and batch argv compatibility checks with one typed command-argument contract that produces the validated request params directly. Keep command-specific semantic validation as small pure functions, generate route matching and help from the same contract, and add coverage that adding a CLI route without parser, help, formatter, batch argv behavior, and request params fails typecheck or a focused registry test.

- [x] Derive extension permission, privilege, and observation checks from protocol-owned command metadata. `packages/protocol/src/metadata.ts` and `packages/protocol/src/registry/*.ts` know security reasons such as `clipboard`, `downloads`, `cookies`, `network-observation`, `page-mutation`, and `page-code-execution`, while `packages/extension/src/manifest.json`, `packages/extension/src/popup.ts`, `packages/extension/src/background-bootstrap.ts`, and `scripts/package-check.ts` hardcode permissions, `<all_urls>` host access, always-on `webRequest` observation, and Firefox data-collection metadata independently. Add a protocol or extension permission manifest module that maps command security reasons to required Firefox permissions, observation lifecycles, and release-gating manifest metadata, generate or validate the extension manifest from that module, and update package checks plus runtime command gates to compare against the generated requirement set. Preserve the MVP full-control model, but make every sensitive capability traceable from command metadata to popup approval, command-time permission checks, network listener activation, manifest permission, Firefox data-collection declaration, and release verification.

- [x] Split the background controller into explicit connection, pairing, negotiation, and request-forwarding services. `packages/extension/src/background-controller.ts` owns storage loading, approval/reset UX messages, native-port lifecycle, reconnect scheduling, pending request tracking, hello negotiation, protocol error state, browser command dispatch, and status JSON. Extract a native connection manager with cancellable reconnect timers, a pairing state service, a negotiated-session service, and a request forwarder with explicit dependencies. Keep `FirefoxCliBackgroundController` as a thin orchestrator, and add tests for races between async pair-token loading, reset, disconnect, reconnect, stop, and stale hello responses.

- [x] Remove implicit log-capture installation from content command handling. `packages/extension/src/content-runtime.ts` already owns explicit global/window log-capture install and dispose, but `packages/extension/src/content-snapshot/router.ts` still calls `installWindowLogCapture(options.document.defaultView)` when no service is injected and discards the returned handle. Require `ContentLogCaptureService` to be supplied by the runtime boundary, delete the handler-level fallback, remove legacy global-state normalization in `packages/extension/src/content-snapshot/log-capture.ts` after a compatibility decision, and add tests proving direct handler calls cannot leak listeners or global console patches.

- [x] Decompose content action execution by action family and DOM effect adapters. `packages/extension/src/content-actions.ts` remains a 545-line mixed surface for command dispatch, element resolution fallback, visibility/enabled checks, file upload decoding, `DataTransfer` construction, mouse/keyboard synthesis, form mutation, scroll mutation, and result rendering. Split it into element-targeted actions, text/editing actions, pointer/keyboard events, selection/upload effects, and scroll effects behind small DOM adapter interfaces. Keep `ActionHandlerMap` exhaustive, but make each action family independently testable without importing unrelated upload, drag, or keyboard machinery.

- [x] Centralize timeout policy and enforce deadlines across native, extension, and script transports. `packages/native-host/src/native-messaging-frame.ts` waits indefinitely while filling a header or body, `packages/native-host/src/native-host-runtime.ts` loops on `reader.read()` until EOF, `packages/extension/src/background-browser-adapter.ts` uses local polling waits, and `scripts/marionette-client.ts` has unbounded frame buffering plus pending commands that remain in `#pending` after timeout. Define named policy objects for CLI-host connect, host-extension negotiation, frame reads, request forwarding, browser waits, Marionette commands, and process stop escalation. Apply those policies by adding read deadlines, draining pending requests with structured timeout errors, extending existing `scripts/script-timing.ts` for script callers, adding an extension-side equivalent for browser waits, and covering stalled streams, timed-out Marionette commands, and socket close with in-flight commands.

- [x] Reject symlink traversal in extension archive, provenance, and package-validation inputs. `scripts/build-extension-archive.ts` walks `dist/extension` with `stat`, which follows symlinks; `scripts/extension-artifact-provenance.ts` recursively reads directory entries without checking canonical containment; and `scripts/package-check.ts` reads the development extension payload with a similar relative-file traversal. Use `lstat` and `realpath` boundary checks before every descent/read, reject symlinks unless explicitly allowlisted, and add tests proving archive generation, provenance hashing, and package validation cannot include files outside the intended extension root.

- [ ] Replace soft JavaScript execution timeouts with an explicit eval and wait-predicate execution contract. `packages/extension/src/eval-executor.ts` and `packages/extension/src/content-wait.ts` execute user-provided code through `new Function`, and their timeouts use `Promise.race`, which cannot interrupt synchronous infinite loops or CPU-bound scripts running in the page context. Decide which execution modes can promise hard timeouts and which page-context modes are non-preemptable, reject or document non-preemptable modes where necessary, move pure evaluation into an abortable isolated runner where possible, and surface structured timeout or unsupported errors instead of allowing an extension command path to hang indefinitely.

- [x] Make content-script injection retry explicit and state-aware. `packages/extension/src/background-browser-adapter.ts` retries every `tabs.sendMessage` failure by injecting `content.js` and sending the message again, while `packages/extension/src/browser-command/content-bridge.ts` turns any remaining failure into a generic injection error. Classify send failures into not-loaded, stale-version, tab-unavailable, restricted-page, permission, and crashed/unloaded states; retry only recoverable not-loaded/stale states; cache successful injection state per tab/document generation; and include the classified cause in protocol errors.

- [x] Close the signed-extension artifact trust boundary before release packaging. `scripts/package.ts` accepts `FIREFOX_CLI_SIGNED_XPI` and `FIREFOX_CLI_SIGNED_XPI_PROVENANCE`, copies them into `dist/package`, and leaves consistency checks to later package validation; `scripts/package-check.ts` checks XPI JAR metadata digests and local provenance but does not verify the PKCS7 signature chain or certificate identity. Validate override files before copy, require regular canonical files and `.xpi` extension, parse provenance with `signedExtensionProvenanceSchema`, verify `packageVersion`, `xpiFile`, and `xpiSha256` against the copied XPI, add a real XPI signature verifier with signer/channel expectations, and then split `scripts/package-check.ts` into package layout, extension payload, signature/provenance, and manifest permission modules as cleanup.

- [ ] Make network observation lazy and target-scoped. `packages/extension/src/background-bootstrap.ts` registers `webRequest` listeners for `<all_urls>` at background startup, and `packages/extension/src/network-tracker.ts` records traffic for every trackable tab up to per-tab caps even when no network command is active. Add a network-observation service with lifecycle ownership, enable listeners only when network commands or network-idle waits require them where Firefox permits, prune inactive target state aggressively, and document any browser API limitation that requires global listener registration.

## Medium Workstreams

- [ ] Move disposable Firefox process discovery and cleanup behind the shared process-management adapter. `scripts/e2e-disposable-firefox.ts` still shells out to `ps`, parses process rows with `parseDisposableFirefoxProcessIds`, and calls `process.kill` directly for cleanup even though `scripts/process-runner.ts` provides tree-aware termination. Create a profile-scoped Firefox process adapter with POSIX and Windows implementations, reuse `ManagedProcess.stop` semantics where possible, and make the E2E cleanup prove it only targets processes for the disposable profile before sending signals.

- [ ] Replace the partial hand-rolled accessibility and visibility model with a documented snapshot semantics layer. `packages/extension/src/content-snapshot/accessibility.ts` implements role inference, accessible names, disabled state, visibility, scrollability, and inline style parsing manually. Extract a snapshot semantics adapter with fixtures for WAI-ARIA name/role cases, hidden subtree behavior, labels, disabled fieldsets, scrollable regions, and iframe descriptors. Prefer a browser-compatible spec-aligned utility if one can run inside the content script; otherwise document the supported subset and keep all deviations covered by tests.

- [ ] Introduce request-scoped target resolution instead of re-querying window state in each handler. `packages/extension/src/browser-handlers/content-routed.ts`, `packages/extension/src/browser-handlers/tabs-windows.ts`, `packages/extension/src/browser-handlers/navigation.ts`, `packages/extension/src/browser-handlers/screenshots.ts`, and `packages/extension/src/browser-handlers/eval-wait.ts` independently call `getOrderedWindows` and resolve targets. Add a browser request context that resolves the target graph once per command, invalidates it after tab/window mutations, and passes resolved targets to handlers. This reduces duplicated browser API calls and makes active-tab race behavior explicit.

- [ ] Validate injected CLI transport responses at the CLI boundary. `packages/cli/src/transport.ts` validates outgoing requests but casts `dependencies.sendRequest` results directly to `ResponseEnvelope<C>`, while the default native-host client validates responses internally. Treat every `sendRequest` implementation as a boundary, parse responses with `parseBoundaryResponseForRequest` or a protocol-session equivalent before formatting, and update CLI tests with malformed injected transport payloads so fakes cannot bypass runtime validation.

- [ ] Tighten protocol constructor and dispatcher typing around validated envelopes. `packages/protocol/src/envelopes.ts`, `packages/protocol/src/typed-dispatch.ts`, and `packages/cli/src/protocol-validation.ts` rely on `as RequestEnvelope<C>`, `as ResponseEnvelope<C>`, and `as never` after Zod validation because the schema registry cannot express command-indexed inference cleanly. Introduce small typed safe-parse wrappers or branded validated payload helpers that return command-indexed parse results, then use them in envelope constructors, dispatcher helpers, and CLI request validation. Keep casts only inside the lowest-level helper and cover it with generic compile-time tests plus runtime parse tests.

- [x] Add explicit Firefox data-collection manifest metadata and validation as a release-gating dependency of the permission metadata workstream. `web-ext lint` warns that `packages/extension/src/manifest.json` is missing `browser_specific_settings.gecko.data_collection_permissions`, and `scripts/manifest-validation.ts` plus `scripts/package-check.ts` do not enforce the metadata. Add the Mozilla-supported declaration that matches the product's data-collection behavior, update manifest schemas and package checks to require it, and document the operator-visible privacy statement in setup or architecture docs if Firefox requires user-facing wording.
